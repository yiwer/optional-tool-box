# M3 实施计划（database-enhancement，PostgreSQL 优先）

> 依据：[00-overview](../design/00-overview.md)、[02-database](../design/02-database-enhancement.md)。方言取向 = **PG-first**（用户 2026-07-07 拍板，解决 R2 开放问题）。
> 蓝本：beacon-database（已完整实现的 ORM 无关数据库 deep module，L1/L2/L3 + 三索引 Handler 注册表 + 双 key 参数源，含 45 条 ADR）。
> 范围：本 M3 交付设计文档 P1+P2 的合体（观测 + 类型系统 + L1/L2/L3 CRUD 内核）；**排除** P3（@AuditField/@Encrypted/分页对接/@Id 雪花策略）与 P4（MySQL 方言/多数据源）。
> 执行方式：subagent-driven development。前序：M0+M1+M2 已在 master（b245450）。环境实测：Docker 29.2.0 可用（Testcontainers PostgreSQL IT 必须真跑）。父 pom 已 pin p6spy 3.9.1、postgresql 42.7.8。

## Global Constraints（全局约束，违反即缺陷）

1. Java 21、Spring Boot 3.5.10；版本由父 pom dependencyManagement 管理，子模块不写版本号（H2 与 org.testcontainers:postgresql 由 Boot BOM 管理，无需父 pin）。
2. 基础包 `cn.code91.toolbox.database`；配置前缀 `toolbox.database.*`；唯一装配类 `ToolboxDatabaseAutoConfiguration`，经 `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports` 注册。
3. **【本仓库唯一的 Result 豁免，最高优先级约束】** 数据访问路径（L1/L2/L3 的执行、Repository 全部 op、RowMapper）**保留 Spring 异常语义，一律不包 `Result<T,E>`**——因为 `@Transactional` 依赖异常传播触发回滚，且 Spring 已提供 `DataAccessException` 统一转译，Result 化会吞掉回滚信号并与生态内所有 ORM 行为相悖（设计文档 02 §4.5）。**仅"配置期校验"fail-fast 抛 `IllegalStateException`**。审查者与实现者都不得把"数据访问不返 Result"当作缺陷；反之，若实现者把 JDBC 执行包进 Result 才是缺陷。
4. 可选依赖（p6spy/postgresql/druid）必须 `<optional>true</optional>`，装配处用 `@ConditionalOnClass(name = "全限定名字符串")` 探测。
5. 每个默认 bean 挂 `@ConditionalOnMissingBean`（Seam）。
6. 复用 facility、禁止重造：`LogUtil`（SQL/慢日志，自动脱敏）、`JsonUtil`（jsonb/json handler 序列化——**只用库 API，不注入 facility bean**，学 beacon ADR-0024 区分"库依赖 vs bean 依赖"）；不自建连接池/事务/JSON。
7. 不使用 Lombok；值对象用 record；注释与 Javadoc 用中文，只写代码本身表达不了的约束（如"alias 不反盖 byJavaType 的原因"、"双 key 的用途"）。
8. 质量门：jacoco BUNDLE INSTRUCTION/LINE ≥ 0.80、BRANCH ≥ 0.70（数据库模块含较多 SDK 胶水与 IT，与前序模块同门槛）；ArchUnit 三规则（包无环；autoconfigure 不被主包依赖；core/spi/dialect 门面不依赖 p6spy 与具体驱动类型——保证缺可选依赖时门面可加载）；`mvn verify` 全绿且测试输出无告警噪音。
9. TDD：先写失败测试再实现，报告含 RED/GREEN 证据。ErrorMapper 类/类型 handler 用真实类型实例断言。
10. git 提交：中文一行主题；结尾带 `Co-Authored-By: Claude Fable 5 <noreply@anthropic.com>`。
11. i18n：P1 无 MessageSource/bundle（与 storage/mail 一致；错误走 Spring 异常，日志为开发者面向中文）。若未来引入，键 `toolbox.database.*` 前缀、示例键只进 src/test/resources（compare 终审 I4 教训）。
12. 数据访问路径的**列名/参数占位一律走命名策略与注解**，不得散落硬编码列名字符串（除测试夹具）。

## 通用设计基线（两个任务共享，均源自 beacon-database 实测）

- **三索引 Handler 注册表**（裁定属 Task 7）：`byJavaType`（查找序列：精确类型 → primitive 装箱 → 枚举 `EnumByNameHandler` 兜底 → assignable 扫描）、`bySqlTypeAlias`（`@Column.sqlType` 精确路由，如 "jsonb"/"uuid"/"inet"）、`byAliasFactory`（参数化 handler 工厂）。**alias handler 不反盖 byJavaType 中的通用 handler**（beacon ADR-0023：同一 Java 类型多种 SQL 表示时避免冲突，alias 仅经别名路由可达）。装配期 `ObjectProvider` 收集应用自定义 handler + customizer → `freeze()`（冻结后 register 抛 `IllegalStateException`，线程安全）。
- **写路径"宽松"**：无 handler 时不报错，回退裸值交给 Spring 默认绑定（容忍未覆盖类型，beacon R3）。
- **双 key 参数源**（裁定属 Task 8）：`AnnotatedParameterSource` 对同一值同时以 Java 字段名（`:userName`）和 SQL 列名（`:user_name`）注册，手写 SQL 与 `SqlBuilder` 生成 SQL 共用一个 ps。
- **列布局缓存**（裁定属 Task 8）：`EntityRowMapper` 按 `ResultSetMetaData` 的列名序列缓存列布局，SELECT 列顺序任意、列子集也支持，相同形状查询复用布局（O(1) 查列下标）。

## Task 7: database 类型系统底座 + 观测 + 装配

必读（按序）：docs/design/00-overview.md §2–§4、docs/design/02-database-enhancement.md 全文、上文"通用设计基线"。

交付物（基础包 `cn.code91.toolbox.database`）：

- `annotation`：`@Table(value)`、`@Id`（P1 仅标记主键，外部赋值，**不做**雪花/自增策略——那是 P3）、`@Column(value, sqlType)`（value 缺省走命名策略；sqlType 为方言类型别名，路由特化 handler）、`@Transient`（不参与 SQL）。
- `naming`：`ColumnNamingStrategy`（seam 接口 + 默认 `CAMEL_TO_UNDERSCORE` 实现：userName→user_name；提供 identity 策略）。
- `spi`：`FieldHandler<J>`（`javaType()` / `read(ResultSet, int)` / `write(J)` / `default sqlTypeAliases()`）、`FieldHandlerRegistryCustomizer`（`customize(registry)`）、`SqlDialect`（`name()` / `limitOffset(sql, limit, offset)` / `registerBuiltins(registry)`——P1 仅 `PostgresDialect` 实现；`SqlBuilder`/`EntityRowMapper` 面向此接口而非 PG 具体类）。
- `registry`：`FieldHandlerRegistry`（三索引 + freeze，见通用基线）+ `registerBuiltins` 静态方法注册标准 handler（String/各数值/Boolean/枚举兜底/日期时间 java.time 全套/byte[]/UUID）。
- `dialect`：`PostgresDialect` + PG 特化 handler（`handler` 子包）：jsonb/json（经 facility `JsonUtil` 库 API，CANONICAL 命名空间序列化）、uuid、inet、text[]/int[] 数组、timestamptz(OffsetDateTime)。
- `observability`：p6spy 集成——`P6spyDataSourceWrappingBeanPostProcessor`（`BeanPostProcessor` 包装 `DataSource`；**`Class.forName("com.p6spy.engine.spy.P6DataSource")` 与构造器在 BPP 构造期解析并缓存**，postProcess 热路径只做 `isInstance`/`newInstance`——beacon ADR-0013 性能教训）。
- `autoconfigure`：`ToolboxDatabaseAutoConfiguration`（`@AutoConfiguration(after = DataSourceAutoConfiguration.class)` + `@ConditionalOnClass(NamedParameterJdbcTemplate.class)` + enabled matchIfMissing=true；beans 全 `@ConditionalOnMissingBean`：`ColumnNamingStrategy`、`SqlDialect`（P1 PG）、`FieldHandlerRegistry`（ObjectProvider 收集 handler+customizer → dialect.registerBuiltins → freeze）；p6spy BPP 为 `static @Bean` + `@ConditionalOnProperty(prefix="toolbox.database.p6spy", name="enabled", havingValue="true")`（**默认 false**，属调试型副作用能力）+ `@ConditionalOnClass(name="com.p6spy.engine.spy.P6DataSource")`）+ `ToolboxDatabaseProperties`（dialect 只读展示为 postgresql / naming-strategy）+ `ToolboxDatabaseP6spyProperties`（enabled）+ imports 文件。
- 模块 pom：compile：`server-facility`、`org.springframework:spring-jdbc`、`spring-boot-autoconfigure`；optional：`p6spy:p6spy`、`org.postgresql:postgresql`；test：`spring-boot-starter-test`、`archunit-junit5`、`com.h2database:h2`、`org.testcontainers:postgresql`、`org.testcontainers:junit-jupiter`。（若某 test 依赖缺版本管理需父 pom 补 pin，按 M2 授权 b 处理并披露。）
- 测试（TDD）：
  - H2/纯 JVM 单测：命名策略双向；注册表三索引查找优先级（精确/装箱/枚举兜底/assignable）、alias 不反盖 byJavaType、freeze 后 register 抛异常、宽松写路径无 handler 回退；标准 handler 读写回环（用 H2 或纯对象）；
  - **Testcontainers PostgreSQL IT**（`*IT`，`@Testcontainers(disabledWithoutDocker=true)`）：PG 特化 handler 真库读写——jsonb（对象↔JSON）、uuid、inet、text[]/int[] 数组、timestamptz；p6spy BPP 真实包装 DataSource 后 SQL 日志产出可验证；
  - 装配矩阵（ApplicationContextRunner）：无 `NamedParameterJdbcTemplate` 类（FilteredClassLoader）→ 全不装配；p6spy 缺类或 enabled=false → BPP 不装配；用户覆盖 `SqlDialect`/`FieldHandlerRegistry`/`ColumnNamingStrategy` 让位；
  - ArchUnit 三规则（约束 8，core/spi/dialect 不得引用 `com.p6spy.**` 与 `org.postgresql.**` 类型）。

验收：`mvn -q verify -pl database-enhancement` 全绿（含 PG IT；jacoco 门达标、analyze 零告警、输出无噪音）。

## Task 8: database CRUD API（L1/L2/L3，建在 Task 7 之上）

必读：Task 7 已落地的 registry/dialect/annotation 代码 + docs/design/02-database-enhancement.md §4.1 + 上文"通用设计基线"的双 key 与列布局缓存。**前置：Task 7 已合入本分支**（其注册表/方言/注解为本任务直接依赖）。

交付物（同基础包，新增三个子包）：

- `params`（L1）：`AnnotatedParameterSource`——`of(entity)` 返回 Spring `SqlParameterSource`（写路径）；**双 key**（Java 字段名 + SQL 列名同注册，见通用基线）；跳过 `@Transient`；值经 `FieldHandler.write` 转换（无 handler 宽松回退裸值）。
- `mapper`（L2）：`SqlBuilder`——`insert/updateById/selectById/deleteById(Class<?>)` 四件套 SQL 文本生成（列名走命名策略/注解，占位符用 SQL 列名，面向 `SqlDialect`）；`EntityRowMapper<T>`——实现 Spring `RowMapper<T>`，经 `FieldHandler.read` 填充实体，**列布局按 ResultSetMetaData 缓存**（见通用基线）。
- `repository`（L3）：`JdbcRepository<T,ID>` 接口（最小 7 op：insert / updateById / findById / deleteById / existsById / count / list(limit,offset)——list 用 `SqlDialect.limitOffset`）+ `PgJdbcRepository<T,ID>` 实现（构造入参 `Class<T>` / `Class<ID>` / `NamedParameterJdbcTemplate` / `FieldHandlerRegistry`；**不做 autoconfigure bean**，由消费方按实体构造，与 beacon 一致）。全部 op **保留 Spring 异常语义，不返 Result**（约束 3）。
- 本任务**不改 autoconfigure**（L3 非单例 bean）；若发现 Task 7 的 registry/dialect 有必须的小调整，最小改动并在报告披露。
- 测试（TDD）：
  - H2/纯 JVM：`SqlBuilder` 四件套 SQL 文本断言（列序、占位符、命名策略生效）；`AnnotatedParameterSource` 双 key（`:userName` 与 `:user_name` 均命中同值）、@Transient 跳过、handler write 生效、无 handler 回退；`EntityRowMapper` 列布局缓存（乱序列/列子集正确映射、相同形状复用）；
  - **Testcontainers PostgreSQL IT**：`PgJdbcRepository` 全 7 op 真库回环（含含 jsonb/uuid/array 字段的实体 insert→findById→updateById→list→count→existsById→deleteById）；L1 手写 SQL 与 L2 生成 SQL 共用同一 `AnnotatedParameterSource` 的双 key 联合验证；
  - 事务语义钉住：一个测试证明 Repository op 抛出的是 Spring `DataAccessException` 家族（不是被包成 Result 的错误值）——防止未来有人误改约束 3。
  - ArchUnit 三规则（复用 Task 7 的规则类或平行新增，覆盖新子包）。

验收：`mvn -q verify -pl database-enhancement` 全绿；随后仓库根 `mvn -q verify` 全绿（不回归 compare/storage/mail）。
